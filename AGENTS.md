## 项目形态

`qq-bot-v2` 是一个接入 NapCat 的 QQ Agent。它接收 QQ 群聊和私聊消息，把入站事实写入 Postgres，并运行 Kagami-style 的单上下文 `BotLoopAgent`。

当前易变实现信息看这些位置：

- `docs/current-state.md`：阶段、启动流程、env vars、模块地图、命令
- `.env.example`：当前环境变量示例和运维说明
- `prisma/schema.prisma`：数据库契约
- `src/index.ts`：启动顺序
- `src/agent/**`：永续上下文 runtime、主循环、工具、compaction
- `src/bot/**`, `src/media/**`, `src/messaging/**`：NapCat 入站、媒体、发送路径

## 两大核心思路

本项目的两条主线是：**永续上下文** 和 **渐进式披露**。

- 永续上下文决定历史怎么稳定存在：LLM 看到的是一个可持久化、可 replay、字节稳定的 `AgentContext` 前缀。已经 append 的历史不因为晚到媒体、工具结果、重跑或 side table 变化而被改写；需要压缩时只走受控 compaction。
- 渐进式披露决定知识和能力怎么便宜进入上下文：常驻 system prompt 只放稳定规则、边界、索引和入口。长风格指南、群口味、数据库 schema、工具手册、外部网页、工作区文件等内容，应通过工具或受控文件路径按需读取。
- 设计新功能时先问两件事：它会不会破坏已 append prefix 的稳定性？它是不是把本可按需读取的大块信息塞进了常驻上下文？

## 永续上下文契约

项目的核心产品契约是：LLM 历史必须稳定、可 replay、低成本扩展。Prompt cache 稳定性是产品特性，不是顺手优化。

硬性不变量：

1. `BotAgentContext` 是 LLM ledger。运行时形态和持久化 snapshot 形态应当一致。不要重新引入“每轮从 N 张表拼 context”的模式。
2. `messages` 是入站事实账本，不是 LLM ledger。入站消息一次性入库；媒体解析后的 `resolvedText` 一次冻结，后续不重写。
3. bot 对外可见的发言必须走 `send_message`。assistant content 是内部推理/历史，不是发送通道。
4. compaction 是正常情况下唯一允许重写 prefix 的路径。保留 assistant tool call 和对应 tool result 的原子性，不要切开。
5. replay 必须确定性。同样输入下，snapshot 里的 message 字节应当跨运行一致。
6. late media、fetch 或 generated bytes 不得改写已经 append 的历史。等待预算耗尽时，使用稳定的降级/冻结表示。
7. 大块外部内容必须走摘要或有边界的工具路径。不要把原始网页、feed、长文件直接塞进主 context。

任何会改动已 append prefix 字节、随手改 tool description、或从可变 side table 重建 prompt 的设计，都应视为 cache/经济性回退，除非任务里明确说明并给出理由。

## 运行模型

- bot 使用一个单一 owned `AgentContext`，在允许的来源之间共享。
- 新事件源必须渲染成确定性的 `user` role message，并通过 event queue / dedup 路径进入 context。
- 跨源知识共享是预期行为；跨源发言仍必须依赖显式 `send_message` target 和 ingress/tool 安全规则。
- system prompt 在启动时构造。修改 prompt 字节会有意让 provider prompt cache 失效，所以 prompt 和 tool description 改动要集中处理。
- `logs/*.ndjson` 是运维旁路日志，不是 Prisma 事实，也不是 AgentContext replay 的一部分。

## LLM 和工具

- Agent chat 当前有 Claude-Code-compatible 路径：`src/agent/claude-code/**`。除非任务明确要求，否则不要改 wire format / cache-control 细节。
- 媒体描述使用 `src/llm/**` 下的 routing provider，和 agent chat client 分离。
- 工具注册集中在 `src/agent/tools/index.ts`。声称某个工具存在前，先查这个文件。
- 每次修改工具注册后，同步检查并更新对应的 system prompt、tool description、测试、文档和旧导出/旧命名残留；不要让 LLM 看到已废弃的工具名或入口。
- 不要针对 prompt 文本写单元测试。prompt 改动用人工 diff、渲染结果抽查、必要时真实对话/工具调用验证；代码解析器或工具契约可以测，但不要把具体措辞锁进测试。
- 建立新能力时遵循“建立时披露”：常驻 system prompt 只放稳定边界、索引和入口；长说明、风格正文、群口味、工具手册、可变数据和外部内容，通过工具、文件工作区或其它按需路径读取。
- Bash 可以作为能力底座或统一交互形态，但不要把裸 shell 暴露给常驻 bot。Bash 类工具必须有命令 allowlist、固定工作区、最小 env、输出/时间上限和审计日志；DB 查询等敏感能力通过专门脚本或 capability 入口复用现有安全层。
- `data/agent-workspace/` 是 agents 自己生产内容的区域，用于 journal、dream、scratch、index、draft 等自组织文件。默认不要把这里的运行时内容当项目源码或事实账本；局部 `.gitignore` 会忽略生成内容，只保留目录说明。除非用户明确要求，否则不要提交该目录下的生成物。
- 有副作用的工具要格外谨慎：`send_message`、图片生成/下载、journal/memory/sticker 工具，以及未来任何会写 DB 或外部服务的工具。
