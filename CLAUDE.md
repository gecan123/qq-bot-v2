本文档是 `qq-bot-v2` 的统一仓库级 agent 指令，适用于 Codex、Claude Code 和其它会读取仓库指令的 agent。`AGENTS.md` 与 `CLAUDE.md` 应保持内容镜像，避免不同 code agent 因加载入口不同而看到不同规则。

本文件只放长期稳定的工作约束和产品契约。当前阶段、默认值、env vars、模块地图、近期决策等易变信息以代码、`.env.example`、`package.json`、`prisma/schema.prisma` 和实际日志为准；如果文档和代码冲突，先相信代码。

## 项目形态

`qq-bot-v2` 是一个接入 NapCat 的 QQ Agent。它接收 QQ 群聊和私聊消息，把入站事实写入 Postgres，并运行 Kagami-style 的单上下文 `BotLoopAgent`。

bot 持有单一永续 `AgentContext`。多源事件 funnel 到同一个上下文里；跨源知识共享是预期行为，跨源发言依赖显式 `send_message` target 和 ingress / tool 安全规则。

当前易变实现信息看这些位置：

- `.env.example`：当前环境变量示例和运维说明
- `package.json`：当前命令和依赖
- `prisma/schema.prisma`：数据库契约
- `src/index.ts`：启动顺序
- `src/agent/**`：永续上下文 runtime、主循环、工具、compaction
- `src/bot/**`, `src/media/**`, `src/messaging/**`：NapCat 入站、媒体、发送路径

## 两大核心思路

本项目的两条主线是：**永续上下文** 和 **渐进式披露**。

- 永续上下文决定历史怎么稳定存在：LLM 看到的是一个可持久化、可 replay、字节稳定的 `AgentContext` 前缀。已经 append 的历史不因为晚到媒体、工具结果、重跑或 side table 变化而被改写；需要压缩时只走受控 compaction。
- 渐进式披露决定知识和能力怎么便宜进入上下文：常驻 system prompt 只放稳定规则、边界、索引和入口。长风格指南、群口味、数据库 schema、工具手册、外部网页、工作区文件等内容，应通过工具或受控文件路径按需读取。
- 设计新功能时先问两件事：它会不会破坏已 append prefix 的稳定性？它是不是把本可按需读取的大块信息塞进了常驻上下文？

## 实验项目策略

本仓库是实验性新项目实现。不要默认为历史数据兼容、迁移成本或旧中间架构让路。

- 当目标架构需要时，可以重新设计、替换或删除旧 runtime、memory、proactive、reply、recovery、admin 等表面。
- 优先选择干净目标模型，而不是 compatibility bridge、dual-write 或长期 legacy adapter。
- 历史数据迁移 / backfill 不是 blocker，除非用户明确要求。
- 不要因为旧概念目前能跑就保留；如果它和目标模型冲突，把它视为可删除。

## 永续上下文契约

项目的核心产品契约是：LLM 历史必须稳定、可 replay、低成本扩展。Prompt cache 稳定性是产品特性，不是顺手优化。Claude / OpenAI prompt cache 都依赖 prefix match；早期消息、system prompt 或 tool description 的无意漂移会让整段前缀重新计费、重新 attention。

硬性不变量：

1. `BotAgentContext` 是 LLM ledger。运行时形态和持久化 snapshot 形态应当一致；不要重新引入“每轮从 N 张表拼 context”的模式。新功能要影响 LLM，只能通过受控 append / compaction 进入 `AgentContext`。
2. `messages` 是入站事实账本，不是 LLM ledger。入站消息一次性入库；媒体解析后的 `resolvedText` 一次冻结，后续不重写。当前 scene invariant：`sceneKind='qq_group'` 时 `groupId` 非空且 `sceneExternalId=''`；`sceneKind='qq_private'` 时 `groupId=null` 且 `sceneExternalId=String(peerId)`。
3. bot 对外可见的发言必须走 `send_message`。assistant content 是内部推理 / 历史，不是发送通道。`send_message` 的 target 必填；群 / 私聊不要混淆。群消息准入主要由 ingress 白名单保证；当前 group ambient 真发还有 `BOT_GROUP_AMBIENT_SEND_IDS` 安全阀，不在白名单时 dry-run。reply 和 private 不受 ambient 白名单影响。
4. compaction 是正常情况下唯一允许重写 prefix 的路径。保留 assistant tool call 和对应 tool result 的原子性，不要切开。
5. replay 必须确定性。同样输入下，snapshot 里的 message 字节应当跨运行一致。启动期 system prompt 会按元数据一次拼装；元数据或 prompt/tool description 改动导致 cache 失效是预期结果，所以要集中改，不要小步频改。
6. late media、fetch 或 generated bytes 不得改写已经 append 的历史。等待预算耗尽时，使用稳定的降级 / 冻结表示；后续 side table 更新不能回头改历史 message。
7. 大块外部内容必须走摘要或有边界的工具路径。不要把原始网页、feed、长文件直接塞进主 context。`fetch_url`、`reddit`、浏览器工具等必须保持实现层截断 / 摘要 / 输出上限；replay 时使用已冻结 tool result，不重新请求外部新值。
8. 产图 / 取图 / 截图类工具可以把压缩预览 image block 写入 `AgentContext`，同时把原图字节放入 OutboundCache 或 artifact 路径供发送链路使用。入站 QQ 图片默认仍以冻结文字描述进入 context。压缩失败时应 graceful fallback 到纯文字结果。
9. 图像句柄契约以代码为准：吃图工具的 `image` 字段使用 `{mediaId}` 或 `{ephemeralRef}`；吐图工具返回 `ephemeralRef`, `dataHash`, `byteSize`, `contentType`, `description` 等稳定字段；`send_message` 对外使用字符串 `imageRef`（`media:N` 或 `ephemeral:<64-hex>`）并在内部转为同一 handle。新增图工具要遵守现有共享 schema 和发送链路。
10. `logs/*.ndjson` 是运维旁路日志，不是 Prisma 事实，也不是 `AgentContext` replay 的一部分。工具调用、fetch、browser action 等日志不能成为 prompt 重建来源。

任何会改动已 append prefix 字节、随手改 tool description、或从可变 side table 重建 prompt 的设计，都应视为 cache / 经济性回退，除非任务里明确说明并给出理由。

## 运行模型

- bot 使用一个单一 owned `AgentContext`，在允许的来源之间共享。
- 新事件源必须渲染成确定性的 `user` role message，并通过 event queue / dedup 路径进入 context。不要插入到历史中段。
- 跨源知识共享是预期行为；跨源发言必须依赖显式 `send_message` target 和 ingress / tool 安全规则。
- system prompt 在启动时构造。修改 prompt 字节会有意让 provider prompt cache 失效，所以 prompt 和 tool description 改动要集中处理。
- idle hint、curiosity tick、background task 完成等运行时事件若进入 LLM，必须走稳定事件渲染或 tool result 路径。

## 数据库和 Prisma

- Prisma client 输出目录是 `src/generated/prisma/`。
- 修改 schema 后运行 `pnpm db:generate`。
- `bot_agent_snapshot` 是单上下文 runtime 的持久化状态。
- `messages` 和 `media` 是入站 / search / media 事实表，不是 `AgentContext` snapshot 的替代品。
- `memory_entries`、`journal_entries`、`sticker_pool` 等自主工具数据默认也不是 prompt 重建来源；是否进入上下文取决于对应工具结果或受控摘要。

## LLM 和工具

- Agent chat 有 Claude-Code-compatible 和 OpenAI-agent 两条主路径。除非任务明确要求，否则不要改 wire format、cache-control 或 provider identity 细节。
- 媒体描述使用 `src/llm/**` 下的 routing provider，和 agent chat client 分离。
- 工具注册集中在 `src/agent/tools/index.ts`。声称某个工具存在前，先查这个文件。
- 每次修改工具注册后，同步检查并更新对应的 system prompt、tool description、测试、文档和旧导出 / 旧命名残留；不要让 LLM 看到已废弃的工具名或入口。
- 不要针对 prompt 文本写单元测试。prompt 改动用人工 diff、渲染结果抽查、必要时真实对话 / 工具调用验证；代码解析器或工具契约可以测，但不要把具体措辞锁进测试。
- 建立新能力时遵循“建立时披露”：常驻 system prompt 只放稳定边界、索引和入口；长说明、风格正文、群口味、工具手册、可变数据和外部内容，通过工具、文件工作区或其它按需路径读取。
- Bash 可以作为能力底座或统一交互形态，但不要把裸 shell 暴露给常驻 bot。Bash 类工具必须有命令 allowlist、固定工作区、最小 env、输出 / 时间上限和审计日志；DB 查询等敏感能力通过专门脚本或 capability 入口复用现有安全层。
- `workspace_bash`、`browser`、`fetch_url`、`fetch_image`、`openbb_cli` 等外部能力必须保持输出上限、超时、审计和风险分级。不要用更宽泛的工具绕过专门工具已有的大小限制、预览压缩、缓存或审计。
- `data/agent-workspace/` 是 agents 自己生产内容的区域，用于 journal、dream、scratch、index、draft 等自组织文件。默认不要把这里的运行时内容当项目源码或事实账本；局部 `.gitignore` 会忽略生成内容，只保留目录说明。除非用户明确要求，否则不要提交该目录下的生成物。
- 有副作用的工具要格外谨慎：`send_message`、图片生成 / 下载、journal / memory / sticker 工具、browser 写操作，以及未来任何会写 DB 或外部服务的工具。
