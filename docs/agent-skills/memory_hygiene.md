# 记忆卫生

`memory` 是长期 Markdown 记忆库，不是聊天摘要，也不是所有消息的备份。它可以记人、群、主题，也可以记 Luna 自己做事形成的经验和线索。

先查:

- 聊到具体人或群，但你不确定以前是否记过偏好、关系、旧话题。
- 对方提到“上次”“之前”“你记得吗”。
- 要对某个人/群做更贴近的回应。
- 接着做一个以前推进过的主题、项目或自审任务。
- 不确定自己之前踩过什么坑、做过什么决定、或形成过什么长期偏好。

再写:

- 对以后仍有用的稳定偏好、事实、关系、禁忌。
- Luna 自己做事时形成的可复用经验、项目线索、设计决定或踩坑记录。
- 用户明确要求你记住。
- 你从多次互动中确认的稳定模式。

不要写:

- 一次性玩笑、临时情绪、普通闲聊。
- 未证实的猜测。
- 长篇原话复制。
- 敏感信息或可能伤害他人的标签化判断。

写入时选择合适 scope:

- `self`: 自己做事、偏好、经验、长期线索。
- `person`: 某个 QQ 用户。
- `group`: 某个群。
- `topic`: 一个主题、项目或长期任务。

写入时用自己的话，一条只记一件事。`topic` 必须使用稳定主题 title，例如项目名、作品名或长期研究问题；不要使用“今日速记”、日期或笼统的 `topic`。查询结果用于自然说话，不要像报数据库。

## 写入前判断

长期记忆不是 append-only 日志。准备写入时：

1. 先用 `memory action=recall` 按自然问题召回 entry；不知道主题文件时再 `search/list`。
2. 已有 entry 只是措辞或事实需要修正：`read` 最新 revision 后 `update_entry`。
3. 两条以上 recent 线索已经形成一个稳定结论：`read` 后 `compact`，用一条 stable 记忆替代原条目。
4. 一条 recent 已经明确跨天有用：`read` 后 `promote_entry`。
5. 确实没有相关事实时才 `write`；完全相同的写入会由 store 去重。

`recent` 是刚观察到、仍可能变化的线索；`stable` 是以后可以直接依赖的长期结论。新事实推翻旧事实时，更新或合并成当前有效结论，不要让两个互相冲突的版本都留作 stable。

## 自主整理

长期记忆开始重复、过时或难检索时再整理，不按固定时间机械执行，也不要为了整理反复生成新的总结。runtime 会在单文件 recent 条目达到阈值、recent 正文过长或只读 review 发现重复/冲突时，把该文件送入共享 maintenance lane；后台整理只会执行受限的 promote/merge/discard 操作，不阻塞当前聊天。

整理顺序:

1. 用 `memory action=review` 找同一文件里的重复、近重复和冲突候选；需要跨文件发现时再 `list/search/recall`。
2. 用 `read` 获取完整 entries 和最新 revision。
3. 同一文件内优先用 `promote_entry/update_entry/compact/delete_entry` 原地整理，让 stable 结论替代 recent 碎片。
4. 只有整个文件都已被另一份明确记忆覆盖时，才用 `delete` 永久删除文件。

`delete` 和 `delete_entry` 不提供回收站。只删除自己已经检查过、并确认被其他记忆覆盖的明确内容；不要顺手清空不相关的 person、group 或 topic 记忆。不要把 stable 与它的 recent 来源同时保留成多份总结。
