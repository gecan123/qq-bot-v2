# Codex-Agent 高上限架构设计说明（中文版）

日期：2026-03-25  
仓库：`qq-bot-v2`  
状态：已确认设计（实施前）

## 1. 目标与范围

本设计面向 QQ 群 `@` 回复场景，目标是构建实验性质、上限更高的智能体架构。

核心目标：
- 将当前“高层业务工具驱动”的回复链路，升级为“通用能力 + agent 自主规划”的架构，以提升问题覆盖与能力上限。

v1 范围内：
- 直接替换当前 `@` 主回复链路。
- 使用 `maxSteps=12` 的多步 agent loop。
- 使用 DB + 网络检索的原子工具集合。
- 提供只读 SQL 能力，并配套严格执行护栏。
- 在终止级失败（超步、全局超时、adapter/系统级故障）时回退单轮回复。
- 提供可用于迭代调参的可观测性。

v1 范围外：
- planner/executor 多智能体拆分。
- 完整离线记忆系统重构。
- 任何写库能力。

## 2. 已锁定决策

以下决策已确认：

1. 架构方向优先“先进 + 更高上限”，不是保守兼容。
2. v1 直接替换主链路，不做 A/B 双入口。
3. v1 能力闭环为“群内历史 + 网络搜索”。
4. agent loop 允许最多 12 步。
5. 初始上下文默认注入最近 20 条群消息。
6. `db_read` 采用受限原生 SQL（非 DSL）。
7. 离线记忆为低优先级，不阻塞 v1。

## 3. 推荐架构

采用“原子能力层 + Agent 自主编排检索”方案。

核心思想：
- 让模型自由决定检索路径与步骤组合。
- 将安全边界、作用域约束、稳定性控制下沉到运行时。
- 避免把推理路径提前固化在高层业务工具里。

### 3.1 组件划分

1. Agent Runtime Adapter
- 责任：对齐模型运行时与项目级 `AgentLlmAdapter` 协议。
- 不承载业务逻辑。

2. Atomic Tools Layer
- 用原子工具替换高层业务工具。
- v1 工具集：
  - `db_schema`
  - `db_read`
  - `web_search`
  - `final_answer`
- 近一步（P1 可选）：`fetch_url`

3. Execution Guardrail Layer
- 在执行前统一做只读 SQL 与策略校验。
- 强制群作用域、超时、行数与输出大小限制。

4. Agent Loop
- 多步工具调用循环，模型通过 `final_answer` 终止。
- 可恢复错误尽量不中断循环。

5. At-Mention Entry
- `@` 消息默认进入新 agent loop。
- 旧单轮逻辑仅保留为兜底。

## 4. 工具契约设计

### 4.1 `db_schema`
用途：
- 给 agent 提供可访问 schema 信息，用于查询规划。

期望输出：
- 可访问表/视图
- 可访问字段
- 约束说明与示例查询

### 4.2 `db_read`
用途：
- 执行单条只读 SQL，并受运行时严格控制。

输入：
- `sql: string`
- `params?: Record<string, string | number | boolean | null>`（调用方可不传；运行时会构造 params 并注入 `group_id`）

输出（结构化）：
- `columns: string[]`
- `rows: unknown[][]`
- `rowCount: number`
- `truncated: boolean`
- `elapsedMs: number`
- 可选 `error` 字段（拒绝或执行失败时）

### 4.3 `web_search`
用途：
- 当群内信息不足时补充外部知识。

输入：
- `query: string`
- 可选 `maxResults?: number`（运行时上限约束）

输出规范：
- 保持稳定结构（如 title/url/snippet）。

### 4.4 `final_answer`
用途：
- 唯一正式终止协议，用于输出用户最终可见文本。

约束：
- payload 固定为 `{ text: string }`（纯文本）。
- 运行时按配置截断最终回答长度。

## 5. SQL 护栏（`db_read`）

以下为 v1 必选项：

1. 语句类型限制
- 仅允许 `SELECT` 或 `WITH ... SELECT` 单语句。
- 拒绝 DDL/DML 及危险关键字。

2. 群作用域硬边界
- 统一契约（唯一口径）：
  - SQL 必须包含 `:group_id` 参数。
  - SQL 必须包含绑定 `:group_id` 的显式群过滤谓词（例如 `group_id = :group_id`）。
  - 运行时负责注入当前群的 `params.group_id`。
  - 运行时不对 SQL 做作用域改写；缺失或不合法直接拒绝。
  - v1 校验器认可的谓词形态：
    - `group_id = :group_id`
    - `<表名或别名>.group_id = :group_id`

3. 运行时限制
- 语句超时。
- 最大返回行数。
- 最大输出字符/字节。

4. 失败语义
- 向模型返回结构化错误；可恢复错误不应直接导致 loop 崩溃。

## 6. Agent Loop 行为

### 6.1 初始化

初始 history（最小但足够）：
- 触发消息文本。
- 若存在，注入 1 层引用上下文。
- 最近 20 条群消息。
- 当前时间与系统边界说明。

原则：
- “宿主给最小种子，agent 自主扩展上下文”。

### 6.2 迭代

每一步：
1. 模型输出工具调用或最终回答。
2. 运行时按声明顺序执行工具。
3. 结果写回 history。
4. 继续下一步直到终止。

### 6.3 终止与降级

正常终止：
- `final_answer`

异常终止：
- 超过最大步数
- 全局超时
- adapter/系统级故障

降级策略：
- 回退到现有单轮回复链路。

## 7. 数据流

1. 输入阶段
- 收到 `@` 消息并构建最小种子上下文。

2. 规划阶段
- agent 动态决定检索策略。

3. 检索阶段
- `db_read` 获取群内证据。
- `web_search` 获取外部证据。

4. 合成阶段
- agent 基于迭代 history 做证据整合。
- 通过 `final_answer` 结束。

## 8. 错误处理与可观测性

每次 run 的必备追踪字段：
- run 状态（`final/fallback/aborted`）
- 终止类型
- 总耗时
- 步数
- 每步工具名
- 每步耗时
- 工具错误详情（若有）
- 行数/输出截断标记

v1 验收指标：
- `final_answer` 终止率
- fallback 率
- timeout 率
- 护栏拒绝率
- 超步中止率
- P50/P95/P99 延迟

量化门槛（上线闸门）：
- `final_answer` 终止率 >= 90%
- fallback 率 <= 15%
- timeout 率 <= 5%
- 超步中止率 <= 8%
- 跨群数据泄露事件 = 0
- P95 端到端延迟 <= 18s
- P99 端到端延迟 <= 30s
- 单次 run 平均输入 tokens <= 12,000

分母定义：
- 除特别说明外，百分比指标均以“进入新 agent loop 的 run（agent-started runs）”为分母，而非全部 `@` 入站事件。

## 9. 交付优先级

### P0（必须）
1. 用新 agent loop 替换 `@` 主链路。
2. 上线原子工具集（`db_schema`、`db_read`、`web_search`、`final_answer`）。
3. 实现 SQL 护栏。
4. 默认注入最近 20 条消息作为种子上下文。
5. 完整 run/step trace 日志。
6. 保留单轮兜底。

### P1（下一轮）
1. 增加 `fetch_url` 做 web 二跳取证。
2. 强化 `db_schema`（示例/注释）。
3. 工具结果升级为“结构化 + 预览文本”双通道。
4. 增加步骤/延迟/fallback 观测面板。

规划说明：
- 本轮 implementation plan 仅覆盖 P0；P1 留作下一周期 backlog。

### Deferred（暂缓）
- P2（记忆双轨、planner/executor、更强 DB 隔离）暂不排期。

## 10. 当前记忆策略

当前结论：
- 记忆不是 v1 阻塞项。
- 在新在线检索架构落地前，不要求先完成记忆重构。

未来方向（本轮不排期）：
- 双轨记忆候选：
  - DB 存结构化事实
  - 文件存叙述性摘要
- 通过检索工具按需读取，不做全量常驻注入。

## 11. 借鉴 OpenClaw 的原则（已本地调研）

以下原则来自开源 agent 体系的通用实践，并已对照 OpenClaw 架构验证：

1. 运行时与上下文治理应解耦。
2. 工具自由度必须配套硬策略管线。
3. `before_tool_call` 类钩子是无破坏演进关键。
4. 必须治理上下文增长（截断、溢出恢复、可压缩接口）。
5. 记忆应“按需检索优先”，而非全量常驻注入。
6. 会话维度串行队列可避免并发导致的上下文竞态。
7. 可观测性应从第一天内建。

说明：
- 本节是设计指导，不作为 v1 额外实现范围门槛。

## 12. 验收标准（实验期）

功能：
- 在不依赖旧高层业务工具的前提下，能完成代表性“群内 + 网络”问答。

能力：
- 多步链路能自然出现（例如 `db_read -> web_search -> final_answer`）。

安全：
- 无跨群泄露。
- 无写库执行。

稳定性：
- fallback 率 <= 15%
- timeout 率 <= 5%
- 超步中止率 <= 8%

性能：
- P95 端到端延迟 <= 18s
- P99 端到端延迟 <= 30s
- 平均输入 tokens <= 12,000

进入 v1.1 规划条件：
- 连续观测 3-7 天，且样本量至少 300 次 `@` run。
- 上述量化门槛全部满足后，再进入下一轮范围。
